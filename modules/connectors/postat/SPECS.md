# Austrian Post (PostAT) integration — specification

Reference for the **postat** connector. Austrian Post
(Österreichische Post) ships through the **Post-Labelcenter (PLC)**
service, a **SOAP / WCF** endpoint on the ondot platform. The connector
supports **shipment creation** (`ImportShipment`) and **shipment
cancellation** (`CancelShipments`); rating is served by karrio's
**universal rating proxy** from a static CSV catalog. There is no
tracking API — the carrier only offers a public web tracking page.

The **vendor source of truth** lives under `vendor/`: the official
`PLC_API_Description.xlsx`, the Postman collection
(`Post Label Center.postman_collection.json`), the full SOAP request
template `request_leer.txt`, a minimal sample `request-example.txt`, a
captured `result-response.txt`, and `services.html` / `services.png`
for the service catalog. The generated Python types
(`karrio/schemas/postat/*.py`) come from the two XSDs under `schemas/`.

## Architecture overview

```
┌─────────────────────────┐
│  Unified shipping model │   karrio ShipmentRequest /
│   (karrio core)         │   ShipmentCancelRequest / RateRequest
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────┐
│  providers/postat       │   Pure data transforms.
│   shipment/create.py    │   Unified model → typed PLC SOAP body,
│   shipment/cancel.py    │   SOAP response → unified model.
│   error.py              │   No HTTP, no side effects.
│   units.py              │   ShippingService, ShippingOption,
│   utils.py              │   TrackingStatus, ConnectionConfig,
│                         │   Settings, SOAP serializer.
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────┐
│  mappers/postat/proxy.py│   HTTP transport only.
│   - create_shipment     │   - POST text/xml SOAP envelope
│   - cancel_shipment     │   - SOAPAction header per operation
│   - get_rates           │   - get_rates delegates to the universal
│                         │     RatingMixinProxy (no PostAT HTTP call)
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────┐
│  Austrian Post PLC      │   SOAP/WCF ShippingService.svc
│  ─────────────────────  │
│  ImportShipment         │   create label + tracking codes
│  CancelShipments        │   void a shipment
│  (no tracking API)      │
└─────────────────────────┘
```

**Key architectural choices:**

- **SOAP/WCF, not REST.** Every call is a `text/xml` SOAP envelope.
  The namespace is `http://post.ondot.at`; the WCF interface is
  `IShippingService`. Each operation carries a distinct `SOAPAction`
  header.
- **Generated XSD types.** `karrio/schemas/postat/plc_types.py`
  (shipment) and `void_types.py` (cancel) are generated by
  `generateDS` from `schemas/postat.xsd` / `schemas/postat_void.xsd`.
  Do not hand-edit — regenerate via `./generate`.
- **Rating is universal, not carrier-native.** `Proxy.get_rates`
  delegates to `karrio.universal.mappers.rating_proxy.RatingMixinProxy`;
  `Mapper.create_rate_request` / `parse_rate_response` call
  `karrio.universal.providers.rating`. Rates come from the static
  `services.csv` catalog projected into `ServiceLevel` objects — PLC
  has no rating endpoint.
- **No tracking integration.** PLC exposes no tracking API; the carrier
  only has a public web page. The connector defines a `TrackingStatus`
  enum but wires no tracking request/parse path.

## Endpoints

There is a **single SOAP service URL** for all operations; the
operation is selected by the `SOAPAction` header, not the path.

| Purpose | Method | URL / SOAPAction |
|---|---|---|
| Default service URL | POST | `https://plc.post.at/Post.Webservice/ShippingService.svc/secure` |
| Create shipment | POST | SOAPAction `http://post.ondot.at/IShippingService/ImportShipment` |
| Cancel shipment | POST | SOAPAction `http://post.ondot.at/IShippingService/CancelShipments` |
| Rates | — | No PLC call — served by the universal rating proxy from `services.csv` |
| Tracking | — | No API. Public page: `https://www.post.at/sv/sendungssuche?snr={}` |

`Content-Type` is `text/xml; charset=utf-8` on every call.

The service URL is a single fixed endpoint, hardcoded in
`utils.SERVER_URL` and **not** operator-configurable. Austrian Post
issues per-customer *credentials* (ClientID / OrgUnit) during onboarding,
not per-customer hosts. Keeping the URL fixed removes the SSRF surface a
tenant-supplied `server_url` would create (SHIP2-1176).

## Authentication

There is **no token exchange**. Credentials are carried inside the SOAP
body, not in HTTP headers. Three values issued by Austrian Post during
onboarding identify the account:

| Setting | Wire field | Notes |
|---|---|---|
| `client_id` | `ClientID` | account/client identifier (test fixture uses `-1`) |
| `org_unit_id` | `OrgUnitID` | organisational-unit id |
| `org_unit_guid` | `OrgUnitGuid` | organisational-unit GUID |

`client_id` is set on `ImportShipmentRowType`; `org_unit_id` /
`org_unit_guid` are set on both the import row and the void row. All
three are required `Settings` fields.

| Tracking | no | — | — |
| Pickup | no | — | — |
| Customs / paperless | no (see [Gaps](#gaps--not-yet-wired)) | — | — |

## SOAP wire shape

`provider_utils.standard_request_serializer` builds the envelope:

- The root `Envelope` and `Body` use the `soapenv` prefix
  (`xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/"`).
- The operation node and its children get the `post` prefix
  (`xmlns:post="http://post.ondot.at"`) via
  `karrio.core.utils.soap.apply_namespaceprefix`.
- Exported with `utils.XP.export(envelope, namespacedef_=…)`.

Two wire-shape facts to know:

- **`original_tagname_` rename.** The generated dataclasses are named
  `ImportShipmentType` / `VoidShipmentType`, but PLC expects the bare
  element names `ImportShipment` / `VoidShipment`. `create.py` /
  `cancel.py` set `import_shipment.original_tagname_ = "ImportShipment"`
  (resp. `"VoidShipment"`) so the serialized element name matches what
  the API accepts.
- **`CustomDataBit1 = False`** is sent on every import row (mirrors the
  vendor sample).

## Services

Service codes map to the PLC `DeliveryServiceThirdPartyID` field. The
`units.py` enum carries **four representative codes** — the actual
codes are configured per account by Austrian Post, so this list is not
exhaustive.

`ShippingService.map(payload.service).value_or_key` resolves the wire
code: a known karrio key yields its numeric `value`; an unknown value
is passed through unchanged (`value_or_key`).

| karrio service key | Wire `DeliveryServiceThirdPartyID` |
|---|---|
| `postat_standard_domestic` | `10` |
| `postat_express_domestic` | `20` |
| `postat_international_standard` | `30` |
| `postat_international_express` | `40` |

The vendor `services.html` documents a wider PLC catalog that an
account may be entitled to (codes are account-specific, several
contract-only):

| PLC ID | Service name | Notes |
|---|---|---|
| `10` | Paket Österreich | standard domestic |
| `45` | Paket Premium International | international |
| `01` | Post Express Österreich | express domestic |
| `46` | Post Express International | express international |
| `28` | Retourpaket | return (contract only) |
| `63` | Retourpaket International | int. return (contract only) |
| `31` | Paket Premium Österreich B2B | B2B |
| `65` | Next Day | contract only |
| `78` | Päckchen M mit Sendungsverfolgung | small tracked package |

## Options (Features)

PLC features ride as `AdditionalInformationRow` entries
(`ThirdPartyID` = feature code, `Value1..4` = parameters) inside
`FeatureList`. The `ShippingOption` enum defines the feature codes:

| karrio option | Wire code | Type | Category | Wire parameters (per vendor docs) |
|---|---|---|---|---|
| `postat_cod` (`cash_on_delivery`) | `COD` | float | COD | Value1=amount, Value2=currency |
| `postat_cod_currency` | `COD_CURRENCY` | str | COD | — |
| `postat_insurance` (`insurance`) | `INS` | float | INSURANCE | Value1=amount, Value2=currency |
| `postat_insurance_currency` | `INS_CURRENCY` | str | INSURANCE | — |
| `postat_signature` (`signature_required`) | `SIG` | bool | SIGNATURE | — |
| `postat_saturday_delivery` (`saturday_delivery`) | `SAT` | bool | DELIVERY_OPTIONS | — |
| `postat_email_notification` (`email_notification`) | `MAIL` | str | NOTIFICATION | Value1=email |
| `postat_sms_notification` | `SMS` | str | NOTIFICATION | Value1=phone |
| `postat_age_verification` | `AGE` | int | SIGNATURE | Value1=minimum age (16 or 18) |
| `postat_label_size` | `label_size` | str | — | label override, see below |
| `postat_paper_layout` | `paper_layout` | str | — | label override, see below |

The unified aliases (`cash_on_delivery`, `insurance`,
`signature_required`, `saturday_delivery`, `email_notification`) point
at the corresponding `postat_*` enum members so standard karrio options
resolve to PLC feature codes.

> **Current limitation:** only `postat_label_size` / `postat_paper_layout`
> are read by `create.py`. The COD / INS / SIG / SAT / MAIL / SMS / AGE
> features are defined in the enum but **not emitted** into a
> `FeatureList` on the request yet — see [Gaps](#gaps--not-yet-wired).

## Label configuration

The label is described on `PrinterObject`. Each field resolves through
a fallback chain (connection config → per-shipment option → default):

| `PrinterObject` field | Source chain | Default |
|---|---|---|
| `LanguageID` | `connection_config.label_format` → `payload.label_type` | — |
| `LabelFormatID` | `connection_config.label_size` → `options.postat_label_size` | `SIZE_100x150` |
| `PaperLayoutID` | `connection_config.paper_layout` → `options.postat_paper_layout` | `LAYOUT_2xA5inA4` |

Allowed enum values (`units.py`):

- `LabelFormat`: `PDF` (`pdf`), `ZPL2` (`zpl2`) — drives `LanguageID`.
- `LabelSize`: `SIZE_100x150` (`100x150`), `SIZE_100x200` (`100x200`).
- `PaperLayout`: `LAYOUT_2xA5inA4` (`2xA5inA4`),
  `LAYOUT_4xA6inA4` (`4xA6inA4`), `LAYOUT_A4` (`A4`) — PDF only.

`PrinterObject.Encoding` (UTF-8 / WINDOWS-1252 for ZPL2) exists in the
XSD/vendor template but is not set by `create.py`.

## Data mapping

### Address — karrio `Address` → PLC `AddressType`

Applies to both `OURecipientAddress` and `OUShipperAddress`.

```
karrio Address                       PLC AddressType
─────────────────                    ───────────────
company_name        ───►             Name1 (or person_name if no company)
person_name         ───►             Name2 (only when Name1 is the company)
address_line1       ───►             AddressLine1   (lib.to_address .street)
address_line2       ───►             AddressLine2
street_number       ───►             HouseNumber    (recipient only)
postal_code         ───►             PostalCode
city                ───►             City
country_code        ───►             CountryID
email               ───►             Email          (recipient only)
phone_number        ───►             Tel1           (recipient only)
```

Shipper address omits `HouseNumber`, `Email`, `Tel1` (only Name1/Name2,
AddressLine1/2, PostalCode, City, CountryID are populated).

### Parcels — karrio packages → `ColloList`

```
karrio Package                       PLC ColloRow
──────────────                       ────────────
weight.KG           ───►             Weight   (kilograms)
length.CM           ───►             Length   (centimetres)
width.CM            ───►             Width
height.CM           ───►             Height
```

One `ColloRow` per parcel; multi-parcel shipments send multiple
`ColloRow` entries under a single `ImportShipmentRowType` (no fan-out).

### References

`payload.reference` → `OUShipperReference1` on the import row.
(`OUShipperReference2` exists in the XSD but is not set.)

## Identifiers

PLC returns tracking codes per parcel in
`ImportShipmentResult.ColloRow[].ColloCodeList.ColloCodeRow`:

```
ColloCodeRow
  ├─ Code                    ───►  tracking_number       (first Code element)
  │                                shipment_identifier    (same value)
  │                                meta.tracking_numbers[] (all Code elements)
  ├─ NumberTypeID            ───►  (not surfaced)         e.g. 213
  └─ OUCarrierThirdPartyID   ───►  (not surfaced)         e.g. OEPAG-DEF
```

The connector uses a **single tier**: the first `Code` is both
`tracking_number` and `shipment_identifier`. That same value is what
the cancel call sends as `VoidShipmentRowType.TrackingNumber`
(populated from `payload.shipment_identifier`).

A shipment is only extracted when `ImportShipmentResult` is present,
at least one `Code` element has text, and there are no error messages —
otherwise `parse_shipment_response` returns `None` plus the messages.

The label comes from `zplLabelData` when present (→ `label_type="ZPL"`),
else `pdfData` (→ `label_type="PDF"`); both are read with
`lib.failsafe`.

## Rating (universal proxy)

The connector has no PLC rating endpoint. Rates are produced by
karrio's **universal rating proxy** over the static catalog in
`services.csv`, loaded at import time by `load_services_from_csv()`
into `DEFAULT_SERVICES`.

`services.csv` columns:
`service_code, service_name, zone_label, country_codes, min_weight,
max_weight, max_length, max_width, max_height, rate, currency,
transit_days, domicile, international`.

Notes from `load_services_from_csv`:

- The carrier `service_code` is mapped to the karrio key via
  `ShippingService.map(service_code).name_or_key`; rows for the same
  service are grouped into one `ServiceLevel` with multiple weight-band
  `ServiceZone`s.
- `transit_days` accepts a range (`"1-3"`); only the lower bound is
  kept as an `int`.
- All `rate` values in the shipped CSV are `0.0` (weight bands /
  country zones are real; prices are placeholders to be filled per
  account).
- `domicile` / `international` are parsed from the literal string
  `"true"`.

If `services.csv` is missing, a single zero-rate
`postat_standard_domestic` `ServiceLevel` is used as a fallback.
`Settings.shipping_services` returns the connection's configured
services if any, else `DEFAULT_SERVICES`.

## Tracking

There is no tracking API. `units.TrackingStatus` exists for a future
mapping but is not consumed anywhere in the connector. For reference,
the defined buckets are:

| karrio status | PLC / DE status codes |
|---|---|
| `pending` | `REGISTERED`, `CREATED`, `DATA_RECEIVED`, `LABEL_PRINTED` |
| `in_transit` | `IN_TRANSIT`, `UNTERWEGS`, `DEPARTED`, `ARRIVED` |
| `out_for_delivery` | `OUT_FOR_DELIVERY`, `IN_ZUSTELLUNG` |
| `delivered` | `DELIVERED`, `ZUGESTELLT`, `POD` |
| `on_hold` | `HELD`, `CUSTOMS` |
| `delivery_failed` | `FAILED`, `NOT_DELIVERED`, `REFUSED` |

End users are pointed at `https://www.post.at/sv/sendungssuche?snr={}`
(the `Settings.tracking_url` template).

## Error parsing

`error.parse_error_response` collects messages from two sources:

1. **SOAP faults** via `karrio.core.utils.soap.extract_fault` — standard
   `<s:Fault>` handling.
2. **PLC `errorMessage` elements** — any `errorMessage` element with
   non-empty text becomes a `Message` with `code="POSTAT_ERROR"`.
   Duplicate text already present in a fault message is skipped.

```
SOAP Response
   │
   ├─► extract_fault(response)          ──► [Message(code from fault)]
   │
   └─► find_element("errorMessage")     ──► [Message(code="POSTAT_ERROR",
        (non-empty, de-duplicated)             message=<text>)]
                                              │
                                              ▼
                                        list[Message]
```

Both shipment-create and cancel parsers run `parse_error_response`
first and only build a success object when no messages are present
(create also requires a tracking `Code`; cancel requires
`VoidShipmentResult.Success == "true"`).

## Gaps / not-yet-wired

These are real in the code today and worth knowing before relying on
the connector:

- **Features not emitted.** `ShippingOption` defines COD / INS / SIG /
  SAT / MAIL / SMS / AGE, but `shipment_request` only reads the two
  label options. No `FeatureList` / `AdditionalInformationRow` is built,
  so those services do not reach PLC yet.
- **No customs payload.** The XSD supports `ColloArticleList`
  (`ColloArticleRow`: ArticleName, HSTariffNumber, CountryOfOriginID,
  ValueOfGoodsPerUnit, …) and address-level `EORINumber` / `VATID`, but
  `create.py` builds neither — international shipments are sent without
  customs article data.
- **No tracking flow** (see above).
- **No pickup flow.**
- **Rates are placeholders.** All `services.csv` rates are `0.0`.
