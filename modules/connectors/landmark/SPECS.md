# Landmark Global integration — specification

Reference for the Landmark Global connector. Landmark Global is an
international cross-border parcel consolidator; this connector speaks its
**XML-over-HTTP v2 API** (`https://api.landmarkglobal.com/v2/<Endpoint>.php`)
for shipment creation, cancellation, returns, and tracking, and serves
**rates from a static, CSV-driven zone catalog** rather than a live rating
API.

The **vendor source of truth** is the Landmark Mercury API docs
(<https://mercurydocs.landmarkglobal.com/docs/api-documentation>). The
XSD schemas, request/response samples, field-length notes, and the tracking
status-code table are captured verbatim in
`vendor/docs-metadata.txt` (plus the PDFs alongside it). The XSDs under
`schemas/*.xsd` are the generation inputs for the typed Python schema
modules.

## Architecture overview

```
┌─────────────────────────┐
│  Unified shipping model │   karrio ShipmentRequest / ShipmentCancelRequest
│   (karrio core)         │   TrackingRequest / RateRequest
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────┐
│  providers/landmark     │   Pure data transforms.
│   shipment/create.py    │   Unified model → typed XML request,
│   shipment/cancel.py    │   typed XML response → unified model.
│   shipment/return_…py   │   No HTTP, no side effects.
│   tracking.py           │
│   error.py              │
│   units.py              │   ShippingService(Name), ShippingOption,
│   utils.py              │   ConnectionConfig, TrackingStatus,
│                         │   TrackingIncidentReason, services.csv loader
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────┐
│  mappers/landmark       │   HTTP transport + settings.
│   proxy.py              │   - POST application/xml
│   settings.py           │   - create/cancel/return → /<API>.php
│                         │   - tracking → concurrent /Track.php (one per number)
│                         │   - rates → universal RatingMixin (no HTTP)
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────┐
│  Landmark Global v2 API │   XML over HTTP, .php endpoints
│  ─────────────────────  │
│  Import.php / Ship.php  │   create shipment (two distinct payloads)
│  Cancel.php             │   cancel / unprocess shipment
│  Track.php              │   tracking history
└─────────────────────────┘
```

**Key architectural choices:**

- **XML, not JSON.** Every request body is serialized via `lib.to_xml`
  with `Content-Type: application/xml`; every response is parsed with
  `lib.to_element`. The typed schema modules under
  `karrio/schemas/landmark/*.py` are `generateDS`-generated from the XSDs
  in `schemas/*.xsd`.
- **Two shipment payloads — Ship vs Import.** A single
  `create_shipment` path emits **either** an `ImportRequest` (→ `Import.php`)
  **or** a `ShipRequest` (→ `Ship.php`), chosen at request-build time and
  carried through `ctx["API"]`. See
  [Ship vs Import](#ship-vs-import--the-two-shipment-payloads).
- **Rates are not fetched from the carrier.** `get_rates` delegates to the
  karrio `universal` `RatingMixinProxy`/`rating_proxy`, which scores the
  request against the static `services.csv` zone table. There is no
  Landmark rate endpoint in this connector.
- **International-only catalog.** Every service is marked
  `international=True, domicile=False` — Landmark Global is a cross-border
  product line.
- **Generated schemas** — `karrio/schemas/landmark/*.py` are generated by
  the `generate` script from `schemas/*.xsd`. Don't hand-edit; regenerate
  with `./bin/run-generate-on modules/connectors/landmark`.

## Data flow

### Shipment creation (one HTTP call)

```
ShipmentRequest                              Landmark v2 API
     │                                            │
     │  shipment_request()                        │
     ├─► lib.to_address(shipper/recipient/        │
     │      return_address)                       │
     ├─► lib.to_packages(parcels)                 │
     ├─► lib.to_shipping_options(...)             │
     ├─► lib.to_customs_info(...)                 │
     ├─► map service → ShipMethod (LGINT*)        │
     ├─► resolve is_import_request                │
     │      (option → connection_config default)  │
     │                                            │
     ├─► ImportRequest  OR  ShipRequest (typed)   │
     ├─► lib.to_xml; ctx{API, label_format, ...}  │
     │                                            │
     │   ─── POST /{API}.php (xml) ──────────────►│   API ∈ {Import, Ship}
     │                                            │   validate / label gen
     │   ◄── <(Import|Ship)Response>              │
     │         <Result><Packages><Package>{       │
     │           TrackingNumber,                  │
     │           LandmarkTrackingNumber,          │
     │           PackageID, BarcodeData,          │
     │           LabelLink / LabelImage }> ───────│
     │                                            │
     ├─► _extract_details:                        │
     │     LandmarkTrackingNumber → tracking_number
     │     PackageID → shipment_identifier        │
     │     LabelImages[].LabelImage → docs.label  │
     │     ShippingCarrier/TrackingNumber → meta  │
     │       (last_mile_*)                        │
     ▼                                            ▼
ShipmentDetails                              (no further call)
```

A shipment is only extracted when the response contains at least one
`TrackingNumber` element; otherwise the parser returns `None` plus any
`Message`s.

### Tracking (one HTTP call per number, concurrent)

```
TrackingRequest                              Landmark Track.php
     │                                            │
     ├─► tracking_request(): one TrackRequest     │
     │     per tracking number, RetrievalType=    │
     │     "Historical"                           │
     │                                            │
     │   ─ lib.run_concurently ─ POST /Track.php ►│   (one call / number)
     │   ◄── <TrackResponse>...<Package><Events>  │
     │         <Event>{Status, DateTime,          │
     │           Location, EventCode}> ───────────│
     │                                            │
     ├─► _extract_details (per number):           │
     │     map EventCode → TrackingStatus         │
     │     map EventCode → TrackingIncidentReason │
     │     normalise events newest-first          │
     │     delivered code wins over latest event  │
     ▼                                            ▼
list[TrackingDetails]                        list[Message]
```

Responses with empty bodies are dropped by the proxy
(`if any(track.strip())`).

## Endpoints

Base URL (prod, hardcoded in `utils.Settings.server_url`):
`https://api.landmarkglobal.com/v2`. There is **no separate sandbox host** —
test mode is signalled in the payload (`<Test>true</Test>`), not by URL.

| Purpose | Method | Path |
|---|---|---|
| Create shipment — Import flow | POST | `{server_url}/Import.php` |
| Create shipment — Ship flow | POST | `{server_url}/Ship.php` |
| Cancel shipment | POST | `{server_url}/Cancel.php` |
| Return shipment | POST | `{server_url}/Import.php` or `/Ship.php` (delegates to create) |
| Tracking | POST | `{server_url}/Track.php` |
| Rates | — | none (served from `services.csv`, see [Rating](#rating-static-csv-driven)) |

The create endpoint path is chosen at runtime from `request.ctx["API"]`
(`"Import"` or `"Ship"`). Tracking sends `Cache-Control: no-cache` in
addition to the XML content type.

Tracking link (for `TrackingInfo.carrier_tracking_link`):
`https://track.landmarkglobal.com/?search={LandmarkTrackingNumber}`.

## Authentication

Per-request credentials embedded in the XML `<Login>` block — there is no
token exchange or OAuth. Every request carries:

| Field | Source (`Settings`) | Notes |
|---|---|---|
| `Login.Username` | `username` | required |
| `Login.Password` | `password` | required (`metadata sensitive` is commented out in `settings.py`) |
| `ClientID` | `client_id` | required; identifies the Landmark client |
| `AccountNumber` | `account_number` | optional; only for clients with multiple account numbers |
| `Test` | `test_mode` | boolean flag in the body; `true` returns mock results |

`region` (default `"Landmark CMH"`) maps to `ShippingLane.Region`, which
links the shipment to a Landmark facility combination assigned per client by
the Integration Manager.

## Supported operations

| Operation | Wired | Implementation |
|---|---|---|
| Rate | yes (static) | `universal` rating mixin over `services.csv` |
| Shipment create | yes | `shipment/create.py` (Import or Ship) |
| Shipment cancel | yes | `shipment/cancel.py` (`Cancel.php`, `DeleteShipment=true`) |
| Return shipment | yes | `shipment/return_shipment.py` (delegates to create) |
| Tracking | yes | `tracking.py` (concurrent per-number `Track.php`) |
| Pickup / Manifest / Document upload | no | not implemented |

## Ship vs Import — the two shipment payloads

`shipment_request` builds **one** of two mutually exclusive request types,
selected by `is_import_request`:

```
is_import_request = options.landmark_import_request.state
                    if set
                    else connection_config.import_request_by_default.state
```

| | `ImportRequest` (`Import.php`) | `ShipRequest` (`Ship.php`) |
|---|---|---|
| Trigger | `landmark_import_request` truthy, or `import_request_by_default` config | default (neither set) |
| `ProduceLabel` | emitted (label is opt-in) | n/a — label always returned |
| `LabelDPI` | not sent | `"300"` when `LabelFormat == "ZPL"` |
| Item `ContentCategory` | nested `ContentCategoryType` (`ReturnCustomsInfo`, `DangerousGoodsInformation`, `ValueAddedServices`) | flat `ContentCategory` string + sibling `ReturnCustomsInfo` |
| Dangerous goods | per-item under `ContentCategory`, gated by `dangerous_goods` option | always `None` |

Both payloads share the same `<Login>`, `ShipTo`, `ShippingLane`,
`ShipMethod`, vendor/fulfillment/return addresses, `Packages`, `Items`, and
`FreightDetails` structure. The Import vs Ship choice and the resolved
`label_format` / `label_encoding` / `produce_label` are stamped on
`Serializable.ctx` so the proxy can route and the parser can label correctly.

Both branches read `Items` from `customs.commodities` when a `customs`
payload is present, else from `packages.items`. `Packages` is omitted
entirely when `fulfilled_by_landmark` is `True` (Landmark fulfils from its
own inventory, so no package dimensions are sent).

## Services

`ShipMethod` wire codes (`provider_units.ShippingService`). All are
international-only. The human-readable name comes from
`ShippingServiceName`; the catalog (zones, weight limits) is built from
`services.csv`.

| karrio service code | Wire `ShipMethod` | Display name | Weight cap |
|---|---|---|---|
| `landmark_maxipak_scan_ddp` | `LGINTSTD` | MaxiPak Scan DDP | 30 KG |
| `landmark_maxipak_scan_ddu` | `LGINTSTDU` | MaxiPak Scan DDU | 30 KG |
| `landmark_minipak_scan_ddp` | `LGINTBPIP` | MiniPak Scan DDP (EU only) | 2 KG |
| `landmark_minipak_scan_ddu` | `LGINTBPIU` | MiniPak Scan DDU (EU & ROW) | 2 KG |
| `landmark_maxipak_scan_ddp_pudo` | `LGINTPUDO` | MaxiPak Scan DDP PUDO | 30 KG |
| `landmark_maxipak_scan_premium_ups_express_ddp` | `LGINTUPSS` | MaxiPak Scan Premium UPS Express DDP | 30 KG |
| `landmark_maxipak_scan_premium_ups_express_ddu` | `LGINTUPSSU` | MaxiPak Scan Premium UPS Express DDU | 30 KG |
| `landmark_maxipak_scan_premium_ups_standard_ddp` | `LGINTUPST` | MaxiPak Scan Premium UPS Standard DDP | 30 KG |
| `landmark_maxipak_scan_premium_ups_standard_ddu` | `LGINTUPSTU` | MaxiPak Scan Premium UPS Standard DDU | 30 KG |
| `landmark_maxipak_scan_pddp` | `LGINTBPMO` | MaxiPak Scan Postal DDP | 30 KG |
| `landmark_minipak_scan_pddp` | `LGINTBPIO` | MiniPak Scan Postal DDP | 2 KG |

`landmark_maxipak_scan_premium` is an alias for
`landmark_maxipak_scan_premium_ups_standard_ddp` (`LGINTUPST`). Weight cap is

so an unrecognised `payload.service` is forwarded verbatim as the
`ShipMethod`.

|---|---|---|---|

### Enumerations

- `WeightUnit`: `LB`, `KG`, `G`
- `DimensionUnit`: `IN`, `CM`
- `LabelFormat`: `PDF`, `JPG`, `GIF`, `BMP`, `ZPL`, `PNG`
- `LabelEncoding`: `LINKS`, `BASE64`, `BASE64COMPRESSED` (the connector

- `PackagingType`: every unified packaging type collapses to the single
  carrier value `PACKAGE`.

## Rating (static, CSV-driven)

There is no Landmark rate API call. `Mapper.create_rate_request` /
`parse_rate_response` defer to karrio's `universal` rating provider, which
matches the unified `RateRequest` against `DEFAULT_SERVICES` — built by
`load_services_from_csv()` from `services.csv`.

CSV columns:
`service_code, service_name, zone_label, country_codes, rate, currency, transit_days`.

Each row becomes a `ServiceZone` (label, rate, country list, transit days)
grouped under a `ServiceLevel` keyed by `service_code`. All shipped rows
carry `rate=0.0, currency=GBP` — the catalog ships **zero-rate placeholders**;
real pricing is configured per connection via `Settings.services`
(`shipping_services`), which overrides `DEFAULT_SERVICES` when populated.

Zones map countries to transit times, e.g. for `LGINTSTD` (MaxiPak Scan DDP):
US (7d), EU Zone 1 `AT,BE,CZ,DE,DK,FR,IE,LU,NL,PL` (8d), EU Zone 2
`ES,FI,HU,IT,PT,SE` (9d), EU Zone 3 `BG,CY,EE,GR,HR,LT,LV,MT,RO,SI,SK` (10d),
Rest of World `AE,AU,CA,HK,NZ,SA,SG` (9d). If `services.csv` is missing, the
loader falls back to one flat-rate zone per service.

## Data mapping

### Address — karrio `Address` → `ShipTo` / `FulfillmentAddress` / `SendReturnToAddress`

```
karrio Address                  Landmark address type
─────────────────               ─────────────────────
company_name or person_name ─►  Name
person_name                 ─►  Attention
address_line1               ─►  Address1
address_line2               ─►  Address2
                                Address3 (always None)
city                        ─►  City
state_code (max 25)         ─►  State        (mandatory for US per vendor)
postal_code                 ─►  PostalCode
country_code                ─►  Country      (ISO-3166 alpha-2)
phone_number                ─►  Phone        (ShipTo only)
email                       ─►  Email        (ShipTo only)
tax_id                      ─►  ConsigneeTaxID (ShipTo only)
```

- `ShipTo` ← `payload.recipient`
- `FulfillmentAddress` ← `payload.shipper`
- `SendReturnToAddress` ← `payload.return_address` (only when present);
  `Code` ← `landmark_return_address_code`
- `VendorInformation` ← the duty-payer address resolved from
  `customs.duty.paid_by` (`sender` → shipper, `recipient` → recipient,
  `third_party` → `customs.duty_billing_address`). Only emitted when a
  `customs` payload is present.

### Vendor / customs identifiers (`VendorInformation`)

From `customs.options`: `VendorLowValueTaxID` ← `low_value_tax_id`,
`VendorCCN` ← `ccn`, `VendorBusinessNumber` ← `business_number`,
`VendorRGRNumber` ← `rgr_number`, `VendorIOSSNumber` ← `ioss`,
`VendorEORINumber` ← `eori_number`.

### Package — karrio package → `Packages.Package`

```
weight_unit / dim_unit ─► WeightUnit / DimensionsUnit
weight                 ─► Weight  (in compatible weight unit)
length/width/height    ─► Length/Width/Height (in compatible dim unit)
reference_number       ─► PackageReference
```

### Item — karrio commodity/item → `Items.Item`

```
sku                          ─► Sku
metadata["LineNumber"]       ─► LineNumber
quantity                     ─► Quantity
value_amount                 ─► UnitPrice
description or title         ─► Description
hs_code                      ─► HSCode (+ ReturnCustomsInfo.HSCode)
origin_country               ─► CountryOfOrigin (+ ReturnCustomsInfo.HSRegionCode)
product_url                  ─► URL
metadata["USMID"]            ─► USMID
```

Import flow additionally maps dangerous-goods item metadata (`UNCode`,
`PackingGroup`, `PackingInstructions`, item weight/volume) under
`ContentCategory.DangerousGoodsInformation` when `options.dangerous_goods`
is set.

Shipment-level money: `OrderTotal` ← `customs.duty.declared_value` or
`options.declared_value`; `ItemsCurrency` ← `options.currency`;
`IsCommercialShipment` ← `"1"` if `customs.commercial_invoice` else `"0"`.

### Shipment response → `ShipmentDetails`

```
<Package> field                       ShipmentDetails
─────────────────────────             ───────────────
LandmarkTrackingNumber  ─►            tracking_number          (customer-facing)
                                      meta.tracking_numbers[]

```

## Tracking

`TrackRequest` is built **one per tracking number** with
`RetrievalType="Historical"` and dispatched concurrently
(`lib.run_concurently`). The parser produces one `TrackingDetails` per
response that carries a `TrackingNumber`.

### Event mapping (`TrackResponse.Package.Events.Event` → `TrackingEvent`)

```
DateTime   ─► date / time / timestamp   (parsed via DATETIME_FORMATS)
Status     ─► description
EventCode  ─► code, status (TrackingStatus), reason (TrackingIncidentReason)
Location   ─► location
```

`DATETIME_FORMATS` accepts `"%m/%d/%Y %I:%M %p"`,
`"%-m/%d/%Y %I:%M %p"`, and `"%Y-%m-%d %H:%M:%S"`.

Events are reordered **newest-first** (Karrio convention) by comparing the
first and last timestamps and reversing if ascending. Status resolution:
if **any** event carries a delivered code, status is `delivered` regardless
of ordering; otherwise the latest (first) event's code decides, defaulting to
`in_transit`. `tracking_number` is the parcel's `LandmarkTrackingNumber`;
`meta.last_mile_carrier` is `ShipmentDetails.EndDeliveryCarrier` unless that
value contains `"routed"` (case-insensitive), in which case it is suppressed.

### Status mapping (`provider_units.TrackingStatus`)

| karrio status | Landmark EventCodes |
|---|---|
| `pending` | `50` (Shipment Data Uploaded) |
| `in_transit` | `60`, `75`, `80`, `100`, `125`, `150`, `155`, `200`, `225`, `250`, `275` |
| `on_hold` | `90` (held for payment), `135` (Customs Issue) |
| `out_for_delivery` | `300` |
| `ready_for_pickup` | `410` |
| `delivery_delayed` | `400` (Attempted delivery), `450` (Re-directed) |
| `delivery_failed` | `900` |
| `delivered` | `500` (delivered), `510` (Proof of Delivery) |

(Vendor codes `550/570` Return received, `800` Claim Issued exist per the
docs but are not mapped.)

### Incident reasons (`provider_units.TrackingIncidentReason`)

| reason | EventCode |
|---|---|
| `consignee_refused` | `900` |
| `consignee_not_available` / `consignee_not_home` | `400` |
| `consignee_incorrect_address` | `450` |
| `customs_delay` / `customs_documentation` | `135` |
| `customs_duties_unpaid` | `90` |

All other reasons are defined with empty code lists (no current mapping).

## Invariants & gotchas

- **Single base URL, payload-driven test mode.** `server_url` is hardcoded
  to the prod host. `Settings.test_mode` is sent as `<Test>` in the body;
  there is no sandbox host. Per the vendor docs, `Test` defaults to `true`
  before account activation and `false` afterwards.
- **`State` is mandatory for US** recipients (vendor note). The connector
  always emits `State` (truncated to 25 chars) and defaults it to `""`.
- **`Region`** (`ShippingLane.Region`) is assigned per client by the
  Integration Manager — default `"Landmark CMH"`. In the `ShipRequest`
  branch the `ShippingLane` block is omitted when `region` is falsy.
- **Vendor info / items are mandatory for consolidators.** Vendor details
  are required when providing 3PL for the client; item details are
  mandatory. The connector emits `VendorInformation` only when a `customs`
  payload exists, and `Items` only when there is at least one item.
- **`fulfilled_by_landmark` suppresses `Packages`.** When set, no package
  dimensions are sent (Landmark fulfils from inventory).
- **Cancel uses `shipment_identifier` as `TrackingNumber`.** The XSD
  enforces exactly one of `Reference` or `TrackingNumber`; the connector
  sends `TrackingNumber` (the `PackageID`/Landmark number) with
  `DeleteShipment=true` and a default reason `"Consignee canceled
  shipment."`.
- **Returns reuse the create path** with `landmark_return_address_code`
  forced to `"RETURN"` (unless already supplied). No dedicated return
  endpoint — `create_return_shipment` calls `create_shipment`.
- **Wire-spelling quirks (verbatim from the API):** the import-request
  option emits the wire field **`InportRequest`** (misspelled in the vendor
  spec); the produce-label config lookup references
  `connection_config.impport_request_produce_label` (double-`p`), a key
  **not declared** in `ConnectionConfig`, so it resolves to an unset option
  and `produce_label` falls back to `False`.
- **Two-tier identifiers.** `LandmarkTrackingNumber` is the customer-facing
  Landmark number (→ `tracking_number`, tracking-link search key);
  `PackageID` is the internal handle (→ `shipment_identifier`). The
  last-mile carrier's own `TrackingNumber` is only surfaced (in `meta`) when
  `Result.ShippingCarrier` is present.
- **Multi-package shipments** return multiple `<Package>` elements; the
  parser keeps the first tracking/identifier as the primary and collects the
  rest into `meta.*` lists. Tracking responses may also carry multiple
  packages (e.g. `...N1`, `...N2` suffixes on the Landmark number).
- **`LabelDPI=300`** is only sent on the Ship flow when `LabelFormat==ZPL`.

## Error parsing

Landmark replies always carry a `<Result><Success>` flag and, on failure,
an `<Errors>` block with one or more `<Error>` children. `error.py` collects
both `Error` and lowercase `error` elements across one or more response
documents and emits a `models.Message` per error that has an `ErrorCode` or
`ErrorMessage`.

```
<(Import|Ship|Cancel|Track)Response>
  <Result><Success>false</Success>
          <ResultMessage>See Errors element for error details</ResultMessage></Result>
  <Errors>
    <Error>
      <ErrorCode>invalid_vas</ErrorCode>          ─► Message.code
      <ErrorMessage>...</ErrorMessage>            ─► Message.message
    </Error>
    ...
  </Errors>
```

- Shipment / cancel parsers run `parse_error_response` first; a shipment is
  only extracted when a `TrackingNumber` is present, and a cancellation only
  when there are zero error messages and `Result.Success` is truthy.
- The tracking parser threads the queried `tracking_number` into each
  `Message.details` so failures correlate back to the requested number.
